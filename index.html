01

Sliding Window

Maintain a moving range over an array/string instead of re-scanning every subarray.

Time O(n) · Space O(k)replaces brute force O(n²)

Trigger keywords

contiguous subarray / substring longest / shortest … that satisfies X window of size k at most / exactly K distinct max sum of k consecutive ✗ not for: non-contiguous subsequences → DP

Visual model

right expands greedily · left chases only when the window breaks a rule — each index enters and leaves once → O(n)

Reference implementation — variable-size window

def length_of_longest_substring(s: str) -> int:
    # LC 3: longest substring without repeating characters
    last_seen = {}                      # char -> most recent index
    left = best = 0

    for right, ch in enumerate(s):    # 1) GROW: right moves every loop
        if ch in last_seen and last_seen[ch] >= left:
            left = last_seen[ch] + 1  # 2) SHRINK: jump left past the duplicate
        last_seen[ch] = right           # 3) UPDATE window state
        best = max(best, right - left + 1)  # 4) RECORD answer while valid
    return best


def max_sum_fixed_k(nums, k: int) -> int:
    # Fixed-size variant: slide, don't recompute the sum
    window = sum(nums[:k])
    best = window
    for i in range(k, len(nums)):
        window += nums[i] - nums[i - k] # add entering, drop leaving element
        best = max(best, window)
    return best
Senior-level tell: "exactly K" problems are usually solved as atMost(K) − atMost(K−1) — interviewers at SDE 3 loops love this reduction (LC 992, 1248).

02

Two Pointers

Two indices moving toward each other (or in tandem) eliminate the nested loop.

Time O(n) · Space O(1)replaces brute force O(n²)

Trigger keywords

sorted array + find pair/triplet pair that sums to target remove duplicates in-place is it a palindrome container / trapping water squash / partition an array O(1) space

Visual model

converging pointers: L and R close in from both ends · n−1 total moves max

Reference implementation — pair sum & 3-Sum skeleton

def two_sum_sorted(nums, target):
    # LC 167: array is sorted — that's the green light
    l, r = 0, len(nums) - 1
    while l < r:
        s = nums[l] + nums[r]
        if s == target:
            return [l, r]
        elif s < target:
            l += 1          # too small → only a bigger left value can help
        else:
            r -= 1          # too big → only a smaller right value can help


def three_sum(nums):
    # LC 15: sort first, fix one element, two-pointer the rest
    nums.sort()
    res = []
    for i in range(len(nums) - 2):
        if i > 0 and nums[i] == nums[i-1]:
            continue                       # skip duplicate anchors
        l, r = i + 1, len(nums) - 1
        while l < r:
            s = nums[i] + nums[l] + nums[r]
            if s < 0:   l += 1
            elif s > 0: r -= 1
            else:
                res.append([nums[i], nums[l], nums[r]])
                l += 1
                while l < r and nums[l] == nums[l-1]:
                    l += 1                 # skip duplicate pairs
    return res
Distinguish from sliding window: two pointers discard the region between/outside them; a window is the answer region. If the question says "contiguous," think window; if it says "pair/triplet in sorted data," think two pointers.

Fast & Slow Pointers

Floyd's cycle detection — two runners at different speeds on a linked structure.

Time O(n) · Space O(1)beats hash-set O(n) space

Trigger keywords

linked list + cycle find the middle of a list happy number / repeating sequence find the duplicate without modifying (LC 287) does the sequence terminate or loop?

Visual model

no cycle → fast hits null first · cycle → relative speed 1 guarantees a meeting inside the loop

Reference implementation — detect cycle & find its entry

def detect_cycle_start(head):
    # LC 142: return the node where the cycle begins (or None)
    slow = fast = head

    # Phase 1 — do they meet?
    while fast and fast.next:
        slow = slow.next            # 1 step
        fast = fast.next.next       # 2 steps
        if slow is fast:
            break                   # collision ⇒ cycle exists
    else:
        return None                 # fast fell off the end ⇒ no cycle

    # Phase 2 — math: dist(head→entry) == dist(meeting→entry)
    slow = head
    while slow is not fast:
        slow = slow.next
        fast = fast.next            # both ×1 now
    return slow                     # cycle entry node


def middle_node(head):
    # LC 876: when fast finishes, slow is at the middle
    slow = fast = head
    while fast and fast.next:
        slow, fast = slow.next, fast.next.next
    return slow
Senior-level tell: LC 287 (Find the Duplicate) treats the array as a linked list (index → value = pointer). Spotting that reduction is exactly what separates an SDE 2 answer from an SDE 3 answer.

Merge Intervals

Sort by start, then sweep once — overlap logic collapses to one comparison.

Time O(n log n) · Space O(n)sort dominates

Trigger keywords

intervals [start, end] overlapping / conflicting meeting rooms / calendar / booking insert / remove an interval minimum rooms / maximum concurrent

Visual model

after sorting, only the LAST merged interval can ever overlap the next one → single pass

Reference implementation

def merge(intervals):
    # LC 56: merge all overlapping intervals
    intervals.sort(key=lambda iv: iv[0])   # sort by START — the whole trick
    merged = [intervals[0]]

    for start, end in intervals[1:]:
        if start <= merged[-1][1]:        # overlaps the last merged one?
            merged[-1][1] = max(merged[-1][1], end)  # extend, don't append
        else:
            merged.append([start, end])    # clean gap → start a new block
    return merged


def min_meeting_rooms(intervals):
    # LC 253: max overlap at any instant — the sweep-line variant
    starts = sorted(iv[0] for iv in intervals)
    ends   = sorted(iv[1] for iv in intervals)
    rooms = best = e = 0
    for s in starts:                       # every start is a +1 event…
        while ends[e] <= s:                # …each finished meeting is a -1
            rooms -= 1; e += 1
        rooms += 1
        best = max(best, rooms)
    return best
Overlap test to memorize: intervals A and B overlap ⟺ A.start ≤ B.end and B.start ≤ A.end. Off-by-one on ≤ vs < ("meetings touching at the boundary") is the most common interview bug here — clarify it out loud.

Binary Search & Search-on-Answer

Any monotonic true/false boundary can be halved — not just sorted arrays.

Time O(log n)on answer: O(n log range)

Trigger keywords

sorted / rotated sorted array required O(log n) find first / last occurrence minimum speed / capacity / days such that … maximize the minimum (or minimize the maximum) huge answer range, cheap feasibility check

Visual model

you're not searching the array — you're searching the boundary of a monotonic predicate

Reference implementation — boundary template + search-on-answer

def first_true(lo, hi, check):
    # Universal boundary template: smallest x in [lo, hi] with check(x)=True.
    # Learn ONE template; never debug off-by-ones again.
    while lo < hi:
        mid = (lo + hi) // 2
        if check(mid):
            hi = mid          # mid works → answer is mid or to the LEFT
        else:
            lo = mid + 1      # mid fails → answer is strictly RIGHT
    return lo                 # lo == hi == boundary


def min_eating_speed(piles, h):
    # LC 875 (Koko): "minimum speed such that…" → binary search the ANSWER
    import math
    def feasible(speed):     # monotonic: faster always still works
        return sum(math.ceil(p / speed) for p in piles) <= h
    return first_true(1, max(piles), feasible)


def search_rotated(nums, target):
    # LC 33: one half of a rotated array is ALWAYS sorted — recurse into order
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if nums[mid] == target: return mid
        if nums[lo] <= nums[mid]:            # left half sorted
            if nums[lo] <= target < nums[mid]: hi = mid - 1
            else:                              lo = mid + 1
        else:                                # right half sorted
            if nums[mid] < target <= nums[hi]: lo = mid + 1
            else:                              hi = mid - 1
    return -1
Senior-level tell: "minimize the maximum" / "maximize the minimum" (LC 410, 1011, 2064) is almost always binary search on the answer with a greedy feasibility check. Saying that sentence out loud scores design points before you write a line.

Monotonic Stack

A stack that only keeps increasing (or decreasing) values — pops answer "next greater/smaller" queries.

Time O(n) · Space O(n)each index pushed/popped once

Trigger keywords

next greater / next smaller element previous greater / smaller days until warmer (waiting / span) largest rectangle in histogram how far can each element "see" remove digits to make smallest number

Visual model

an element waits on the stack until the FIRST item that beats it arrives — the pop moment IS the answer

Reference implementation — next greater / daily temperatures

def daily_temperatures(temps):
    # LC 739: for each day, how many days until a warmer one?
    res = [0] * len(temps)
    stack = []                              # indices of a DECREASING run

    for i, t in enumerate(temps):
        # today's temp resolves every colder day still waiting
        while stack and temps[stack[-1]] < t:
            j = stack.pop()
            res[j] = i - j                  # answered at pop time
        stack.append(i)                     # today now waits its turn
    return res                              # unresolved indices stay 0


def largest_rectangle(heights):
    # LC 84: a pop means we finally know the bar's LEFT and RIGHT walls
    stack, best = [], 0
    for i, h in enumerate(heights + [0]):   # sentinel 0 flushes the stack
        while stack and heights[stack[-1]] >= h:
            height = heights[stack.pop()]
            left = stack[-1] if stack else -1
            best = max(best, height * (i - left - 1))
        stack.append(i)
    return best
Choosing direction: need next greater? Keep a decreasing stack. Need next smaller? Keep an increasing stack. The stack's order is always the opposite of what you're looking for.

Heap / Top-K

Keep only the k items that matter; the heap evicts the rest in log k time.

Time O(n log k) · Space O(k)vs full sort O(n log n)

Trigger keywords

k largest / k smallest / kth element k most frequent / k closest median of a data stream merge k sorted lists repeatedly take the min / max (scheduler)

Visual model

counter-intuitive but key: kth LARGEST uses a MIN-heap (and vice versa) — the root is the weakest member of the elite club

Reference implementation — kth largest & two-heap median

import heapq

def kth_largest(nums, k):
    # LC 215: min-heap of size k — root is always the kth largest
    heap = nums[:k]
    heapq.heapify(heap)                     # O(k)
    for x in nums[k:]:
        if x > heap[0]:                     # beats the weakest member?
            heapq.heapreplace(heap, x)      # pop root + push, one log k op
    return heap[0]


class MedianFinder:
    # LC 295: two heaps split the stream around the median
    def __init__(self):
        self.lo = []   # MAX-heap (negated) — smaller half
        self.hi = []   # MIN-heap            — larger half

    def addNum(self, num):
        heapq.heappush(self.lo, -num)                    # 1) into smaller half
        heapq.heappush(self.hi, -heapq.heappop(self.lo)) # 2) rebalance max → min
        if len(self.hi) > len(self.lo):                  # 3) keep lo ≥ hi in size
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2
Interview nuance: for "kth largest," mention quickselect (average O(n)) as the alternative and explain why you'd still pick the heap for a stream or when k ≪ n. Python's heapq is min-heap only — negate values for max-heap behavior.

Breadth-First Search

Explore in rings. The first time BFS touches a node IS the shortest unweighted path.

Time O(V + E) · Space O(V)grid: O(m·n)

Trigger keywords

shortest path with unit / equal weights minimum number of steps / moves level order traversal spreads / rotting / infection per minute nearest exit / gate / 0 ✗ weighted edges → Dijkstra (heap-BFS)

Visual model

BFS = ripples in a pond — equal distance frontier expands one ring per iteration

Reference implementation — multi-source grid BFS

from collections import deque

def oranges_rotting(grid):
    # LC 994: rot spreads from ALL rotten oranges simultaneously
    # → seed the queue with EVERY source (multi-source BFS)
    rows, cols = len(grid), len(grid[0])
    q, fresh = deque(), 0

    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == 2: q.append((r, c, 0))  # (cell, minute)
            elif grid[r][c] == 1: fresh += 1

    minutes = 0
    while q:
        r, c, t = q.popleft()
        minutes = max(minutes, t)
        for dr, dc in ((1,0), (-1,0), (0,1), (0,-1)):
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols and grid[nr][nc] == 1:
                grid[nr][nc] = 2        # mark visited ON ENQUEUE — avoids dupes
                fresh -= 1
                q.append((nr, nc, t + 1))

    return minutes if fresh == 0 else -1
Senior-level escalations: weighted edges → Dijkstra (swap deque for a min-heap). Edge weights only 0/1 → 0-1 BFS with a deque. Two ends known → bidirectional BFS (LC 127 follow-up). Knowing the ladder is an SDE 3 signal.

DFS & Backtracking

Walk the decision tree: choose → explore → un-choose. Prune branches that can't win.

Time O(b^d) worst caseislands: O(m·n)

Trigger keywords

generate all combinations / permutations / subsets all possible ways / paths / partitions place pieces with constraints (N-Queens, Sudoku) word search in a grid count islands / connected regions / flood fill small n (n ≤ ~20) — exponential is expected

Visual model

the recursion IS the tree — path = current choices · leaves (or every node) = results · pruning makes it tractable

Reference implementation — backtracking template + islands

def subsets(nums):
    # LC 78: the universal backtracking skeleton
    res, path = [], []

    def backtrack(start):
        res.append(path[:])             # record a copy of current state
        for i in range(start, len(nums)):
            path.append(nums[i])        # 1) CHOOSE
            backtrack(i + 1)            # 2) EXPLORE (start=i+1 → no reuse)
            path.pop()                  # 3) UN-CHOOSE — restore state!

    backtrack(0)
    return res
    # permutations: loop ALL i, track a `used` set
    # combination sum: recurse with i (not i+1) to allow reuse


def num_islands(grid):
    # LC 200: DFS flood fill — count connected components
    rows, cols = len(grid), len(grid[0])

    def sink(r, c):
        if not (0 <= r < rows and 0 <= c < cols) or grid[r][c] != "1":
            return
        grid[r][c] = "0"                # mark visited by sinking the land
        for dr, dc in ((1,0), (-1,0), (0,1), (0,-1)):
            sink(r + dr, c + dc)

    count = 0
    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == "1":
                count += 1              # new island found…
                sink(r, c)              # …erase all of it
    return count
Complexity script: subsets → O(2ⁿ·n), permutations → O(n!·n), and say it unprompted. The single most common backtracking bug is forgetting path.pop() or appending path instead of path[:].

Dynamic Programming

Backtracking's decision tree, but overlapping subproblems get solved once and cached.

Time O(states × transitions)often O(n) or O(n·m)

Trigger keywords

count the number of ways maximum / minimum … over a sequence of choices can you reach / partition / form target longest increasing / common subsequence take or skip · rob or not rob edit / convert one string to another

Visual model

if your backtracking tree solves the same subproblem twice, you've found a DP — cache it

Reference implementation — 1D, knapsack, 2D strings

def rob(nums):
    # LC 198 — 1D DP with O(1) space (rolling variables)
    prev2 = prev1 = 0          # dp[i-2], dp[i-1]
    for x in nums:
        prev2, prev1 = prev1, max(prev1, prev2 + x)  # skip vs rob
    return prev1


def can_partition(nums):
    # LC 416 — 0/1 knapsack: can a subset hit sum/2?
    total = sum(nums)
    if total % 2: return False
    target = total // 2
    dp = [True] + [False] * target   # dp[s] = "sum s is reachable"
    for num in nums:
        for s in range(target, num - 1, -1):  # REVERSE → each item used once
            dp[s] = dp[s] or dp[s - num]
    return dp[target]
    # unbounded knapsack (LC 322 Coin Change): iterate s FORWARD instead


def longest_common_subsequence(a, b):
    # LC 1143 — the 2D string-DP archetype (edit distance is a cousin)
    m, n = len(a), len(b)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if a[i-1] == b[j-1]:
                dp[i][j] = dp[i-1][j-1] + 1          # match → extend diagonal
            else:
                dp[i][j] = max(dp[i-1][j], dp[i][j-1]) # drop one char
    return dp[m][n]
SDE 3 framing: always derive the brute-force recursion first, point at the repeated subproblems, then add the cache. Jumping straight to a table you memorized reads as pattern-matching; deriving it reads as engineering. Mention the space optimization (2D → two rows) as a closer.

Topological Sort

Order a DAG so every dependency comes before its dependents. Kahn's = BFS by in-degree.

Time O(V + E) · Space O(V)also detects cycles free

Trigger keywords

prerequisites / course schedule dependency / build order task A must come before task B is the ordering possible? (cycle check) alien dictionary / derive an ordering from constraints

Visual model

think "peeling an onion": repeatedly remove whatever has nothing pointing at it

Reference implementation — Kahn's algorithm

from collections import deque, defaultdict

def find_order(num_courses, prerequisites):
    # LC 210: return any valid course order, or [] if impossible
    graph = defaultdict(list)
    indeg = [0] * num_courses

    for course, pre in prerequisites:
        graph[pre].append(course)       # edge: pre → course
        indeg[course] += 1              # course gained a dependency

    # start with everything that has NO unmet dependencies
    q = deque(c for c in range(num_courses) if indeg[c] == 0)
    order = []

    while q:
        node = q.popleft()
        order.append(node)              # safe to take: deps all done
        for nxt in graph[node]:
            indeg[nxt] -= 1             # one dependency satisfied
            if indeg[nxt] == 0:
                q.append(nxt)           # fully unblocked → schedule it

    # couldn't place every node ⇒ a cycle blocked the rest
    return order if len(order) == num_courses else []
Real-world hook (great for SDE 3 rounds): this is literally how build systems (Bazel, Make), package managers, and Airflow DAGs schedule work. Connecting the algorithm to systems you've operated is exactly the seniority signal interviewers look for.

Union-Find (Disjoint Set)

Near-O(1) "are these connected?" and "merge these groups" — connectivity that evolves over time.

Time O(α(n)) ≈ O(1) per opwith compression + rank

Trigger keywords

groups / accounts that merge are A and B connected? (many queries) number of connected components / provinces redundant connection / edge that creates a cycle edges arrive over time (dynamic connectivity) ✗ need paths, not just connectivity → BFS/DFS

Visual model

same root = same group · union = pointing one root at the other — that's the whole data structure

Reference implementation — DSU with both optimizations

class DSU:
    def __init__(self, n):
        self.parent = list(range(n))    # everyone starts as their own root
        self.rank = [0] * n             # tree-height upper bound
        self.components = n

    def find(self, x):
        while self.parent[x] != x:
            # path compression: grandparent hop flattens as we walk
            self.parent[x] = self.parent[self.parent[x]]
            x = self.parent[x]
        return x

    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return False                # already connected — a cycle edge!
        if self.rank[ra] < self.rank[rb]:
            ra, rb = rb, ra             # union by rank: small under big
        self.parent[rb] = ra
        if self.rank[ra] == self.rank[rb]:
            self.rank[ra] += 1
        self.components -= 1
        return True


def find_redundant_connection(edges):
    # LC 684: the first edge whose union "fails" closes a cycle
    dsu = DSU(len(edges) + 1)
    for a, b in edges:
        if not dsu.union(a, b):
            return [a, b]
Union-Find vs DFS: if the graph is fully given up front, DFS counts components just fine. Reach for DSU when edges arrive incrementally or you must answer many connectivity queries interleaved with merges — say this trade-off out loud.

Prefix Sum

Precompute cumulative totals once, then answer any range-sum query in O(1).

Build O(n) · Query O(1)vs re-summing O(n)/query

Trigger keywords

sum of a subarray / range sum many repeated queries over ranges count subarrays that sum to k divisible by k / equal 0s and 1s running total · balance over time ✗ if the array mutates a lot → Fenwick/segment tree

Visual model

one subtraction replaces a whole loop · the "sum to k" trick stores prefix counts in a hashmap as you go

Reference implementation — range query & the hashmap variant

class NumArray:
    # LC 303: build once, answer unlimited range-sum queries in O(1)
    def __init__(self, nums):
        self.prefix = [0]              # leading 0 removes edge cases
        for x in nums:
            self.prefix.append(self.prefix[-1] + x)

    def sumRange(self, i, j):
        # sum of nums[i..j] inclusive
        return self.prefix[j + 1] - self.prefix[i]


def subarray_sum_equals_k(nums, k):
    # LC 560: count subarrays summing to k — the must-know variant
    # key insight: a subarray sums to k  ⟺  running − k seen before
    from collections import defaultdict
    count = running = 0
    seen = defaultdict(int)
    seen[0] = 1                        # empty prefix: one way to make sum 0

    for x in nums:
        running += x
        count += seen[running - k]     # how many earlier prefixes complete a k-sum?
        seen[running] += 1            # record this prefix for future indices
    return count
The pattern hides in disguises: "binary subarray with equal 0s and 1s" (map 0→−1), "subarray divisible by k" (store running % k), "find pivot index" — all the same prefix machinery. Recognizing a sum/balance question as prefix-sum is the whole skill. The hashmap variant (LC 560) is the single most common form in real interviews.

In-Place Linked List Reversal

Re-point each node backward with three moving references — no extra memory.

Time O(n) · Space O(1)vs array copy O(n) space

Trigger keywords

reverse a linked list reverse a sublist [m, n] reverse in groups of k O(1) space required on a list reorder / swap nodes by relinking

Visual model

three pointers walk the list once · the only trick is saving next_node BEFORE you overwrite ptr.next

Reference implementation — full reverse, then groups of k

def reverse_list(head):
    # LC 206: the three-pointer dance — memorize this cold
    prev, ptr = None, head
    while ptr:
        next_node = ptr.next        # 1) SAVE the rest of the list
        ptr.next = prev             # 2) FLIP this node's arrow backward
        prev = ptr                  # 3) advance prev…
        ptr = next_node             # 4) …and ptr by one
    return prev                     # prev is the new head


def reverse_k_group(head, k):
    # LC 25: reverse every block of k nodes, leave a short tail as-is
    # 1) check there are k nodes left before touching anything
    node, count = head, 0
    while node and count < k:
        node, count = node.next, count + 1
    if count < k:
        return head               # fewer than k → leave this block alone

    # 2) reverse the first k nodes; afterwards `head` becomes the block's tail
    prev, ptr = None, head
    for _ in range(k):
        nxt = ptr.next
        ptr.next = prev
        prev = ptr
        ptr = nxt
    # prev = new block head · ptr = head of the next block
    # 3) recurse on the remainder and stitch it onto this block's tail
    head.next = reverse_k_group(ptr, k)
    return prev                     # new head of this reversed block
Pointer hygiene: in the k-group reverse, seeding prev with the already-reversed remainder is what stitches the blocks together in one pass. For "reverse sublist [m,n]" (LC 92), use a dummy head and a prev sentinel before position m so the splice stays clean. The universal bug here is losing the rest of the list — always save next first.
